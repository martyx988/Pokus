# T12: App-Facing Supported Universe Retrieval

## Purpose
Expose only app-visible supported instruments and their support/signal-readiness state to mobile clients.

## Scope
- Implement or extend the supported universe retrieval endpoint/read model.
- Filter by supported exchange where requested.
- Return supported instruments only.
- Include support state and signal-readiness state needed by the app.
- Reject unsupported exchange requests.

## Allowed Changes
- Public mobile API endpoint/read model for supported universe retrieval.
- Serializer/resource shaping for supported instruments.
- Authorization/API-key checks only as required by the existing Milestone 1 API foundation.
- Focused API and filtering tests.

## Forbidden Changes
- Do not expose candidate listings, excluded instruments, unsupported duplicates, or internal ranking/audit details.
- Do not implement readiness, current-day price, historical price, or signal endpoints.
- Do not make admin/operator data public.
- Do not create publication behavior.

## Inputs
- T1 exchange scope.
- T6 supported universe state.
- T9 selected supported listing metadata.
- T11 discovery workflow output.

## Relevant Context
The public mobile API must expose only supported, app-visible instruments. Excluded and unsupported candidates must be hidden. Signal generation may be unavailable until enough history exists, so the response must distinguish supported price visibility from signal readiness.

## Expected Output
Mobile clients can retrieve the supported universe for configured launch exchanges without seeing unsupported candidates.

## Acceptance Criteria
- Requests for a supported launch exchange return only app-visible supported instruments for that exchange.
- Requests for unsupported exchanges are rejected or return a controlled unsupported-exchange response.
- Excluded, degraded-internal-only, removed, historical-only, and non-selected duplicate candidates are not returned unless their state is explicitly app-visible by the domain rules.
- The response includes support state and signal-readiness state.
- Results are stable across repeated discovery workflow runs.

## Test Requirements
- API tests for supported exchange filtering.
- API tests for unsupported exchange rejection.
- API tests proving excluded and candidate-only records are hidden.
- API tests for signal-readiness placeholder output.

## Dependencies
- T1.
- T6.
- T9.
- T11.

## Stop Condition
Stop when app-facing universe retrieval returns only supported instruments with the required readiness state and test coverage.
