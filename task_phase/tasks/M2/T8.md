# T8: Exchange Activity Priority Computation

## Purpose
Compute and store exchange activity priority used as the final tie-breaker in primary/best-listing selection.

## Scope
- Compute each exchange priority from trailing 60 trading-day average total traded value across supported/candidate listings when candidate turnover inputs are available.
- Normalize traded value to USD/EUR-equivalent only through existing currency/FX assumptions or explicit placeholder fields; flag missing normalization data rather than inventing rates.
- Store the computed rank and computation metadata.
- Provide a monthly recomputation entry point without wiring unrelated scheduling behavior.

## Allowed Changes
- Universe/ranking service module.
- Exchange activity priority fields or related model/table.
- Job registration or command entry point for monthly recomputation if Milestone 1 job framework exists.
- Focused ranking tests.

## Forbidden Changes
- Do not implement provider validation, historical price loading, or FX-rate providers.
- Do not hardcode permanent exchange priority as the ranking source.
- Do not choose supported listings in this task.
- Do not change publication or readiness behavior.

## Inputs
- T1 launch exchange records.
- T5 candidate listing turnover/traded-value fields.
- T3 calendar service for trading-day windows where needed.

## Relevant Context
The global primary/best-listing policy ranks candidates by home exchange, highest turnover, then automatically derived exchange activity priority. Activity priority is based on trailing 60 trading-day average total traded value across supported/candidate listings, normalized for cross-market comparison, recomputed during validation/discovery and initially monthly.

## Expected Output
Exchange activity priority can be computed, stored, and recomputed deterministically from available candidate turnover inputs.

## Acceptance Criteria
- Priority ranks are deterministic for exchanges with sufficient turnover data.
- Missing turnover or currency-normalization data is surfaced as a controlled reason, not silently guessed.
- Recomputing priority updates ranks idempotently.
- A monthly recomputation entry point exists for later scheduler wiring or is registered with the existing job system if appropriate.

## Test Requirements
- Unit tests for priority ordering from candidate traded values.
- Tests for missing turnover and missing normalization inputs.
- Idempotency test for recomputation.
- Calendar-window test if trading-day window calculation is implemented here.

## Dependencies
- T1.
- T3.
- T5.

## Stop Condition
Stop when exchange activity priority is computed and persisted without selecting supported listings.
