## T21: Admin Scope Configuration for Exchanges and Instrument Types

### Purpose
Enable admin to define supported exchanges and supported instrument types used by universe workflows.

### Scope
Add or finalize private admin configuration surface and persistence for exchange/type support toggles.

### Allowed Changes
- Admin configuration modules under `project/`
- Domain/config models tied only to support-scope flags
- Private admin API/command handlers and focused tests for scope updates

### Forbidden Changes
- Do not implement discovery crawling, listing ranking, or exclusion lifecycle logic.
- Do not add app-facing universe endpoints.
- Do not add provider price ingestion.

### Inputs
- `task_phase/roadmap_plan.md`
- `design_phase/architecture.md`
- `design_phase/product_spec.md`
- Milestone 1 admin/auth foundations

### Relevant Context
Milestone 2 starts from admin-selected exchanges and instrument types. Product spec FR-1/FR-2 requires this scope to be configurable and auditable before discovery runs.

### Implementation Budget
- Production files: max 2-4 closely related admin/config files.
- Behavior count: 1 behavior (persist and update support scope).
- Model/API/workflow count: 0-1 config model adjustment, 1 private admin surface, 0 discovery workflows.
- Test scope: focused admin scope set/get and validation of allowed exchange/type identifiers.

### Expected Output
Admin can persist supported exchange and instrument-type scope used by downstream worker workflows.

### Acceptance Criteria
- Private admin action exists to set supported exchanges.
- Private admin action exists to set supported instrument types.
- Persisted scope is readable by worker-side components.
- Unauthorized callers cannot mutate scope.

### Test Requirements
- Admin auth/permission test for scope mutation.
- Validation tests for unsupported exchange/type identifiers.
- Persistence/readback test for saved scope.

### Dependencies
- M1 foundations.

### Stop Condition
Stop when scope configuration is persisted, validated, and readable for worker usage.

### Completion Format
Implementation agent must finish task file with:
- `### Status`
- `Done.`
- `### Completion Summary`
- concise summary of completed work
