# T2: Restricted Admin Scope Configuration Surface

## Purpose
Allow an authorized admin/operator to view and change supported exchange and instrument-type scope.

## Scope
- Expose configuration controls through the existing private admin/operator foundation.
- Support enabling/disabling configured exchanges and instrument types.
- Record admin actor, timestamp, and reason or note where the Milestone 1 admin audit pattern supports it.
- Validate that only known launch scope records can be configured in this milestone.

## Allowed Changes
- Existing admin/operator configuration modules.
- Forms, serializers, handlers, or views for exchange and instrument-type configuration.
- Admin audit/log hooks related only to scope changes.
- Focused authorization and configuration tests.

## Forbidden Changes
- Do not create a public admin surface.
- Do not implement exchange validation, provider discovery, price ingestion, or publication.
- Do not allow arbitrary unsupported exchanges or crypto scope to be admitted.
- Do not bypass existing Milestone 1 authentication or authorization boundaries.

## Inputs
- T1 launch scope configuration persistence.
- Existing Milestone 1 admin/operator access control.

## Relevant Context
Admin/operator capabilities must be private and restricted. Milestone 2 requires admin configuration for supported exchanges and instrument types; later provider validation and ingestion depend on this scope. Public mobile clients must never modify scope.

## Expected Output
An authorized admin can inspect and update the enabled scope for launch exchanges and launch instrument types.

## Acceptance Criteria
- Authorized admin users can view all launch exchange and instrument-type scope records.
- Authorized admin users can enable or disable supported scope values.
- Unauthorized users cannot read or mutate scope configuration.
- Scope changes create the expected audit/log record if the project already has admin audit support.
- Invalid or unknown scope identifiers are rejected.

## Test Requirements
- Authorization tests for allowed and forbidden users.
- Validation tests for unknown exchange and instrument-type identifiers.
- Audit/log assertion for successful scope changes where supported by the existing foundation.

## Dependencies
- T1.

## Stop Condition
Stop when admin scope configuration works through the private admin/operator boundary and is covered by focused tests.
