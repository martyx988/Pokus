# T5: Candidate Listing and Identifier Persistence

## Purpose
Persist discovered candidate listings and stable identifier records for later support decisions.

## Scope
- Store candidate listing records discovered within configured exchange/type scope.
- Link candidates to exchange and instrument-type configuration.
- Store symbol, name, listing currency, home exchange marker where available, turnover/traded-value inputs where available, provider/source name, and discovery timestamp.
- Store stable provider or reference identifiers when available, such as provider IDs, ISIN, FIGI, or similar reference identifiers.

## Allowed Changes
- Domain models for candidate listings and identifier records.
- Database migrations for candidate listing and identifier persistence.
- Repository/service helpers for idempotent candidate upserts.
- Focused persistence tests.

## Forbidden Changes
- Do not create supported universe state in this task.
- Do not implement best-listing ranking.
- Do not create universe-change audit records except low-level persistence timestamps if already standard.
- Do not store price records or provider price attempts.

## Inputs
- T1 launch scope records.
- T4 normalized discovery candidate contract.

## Relevant Context
The product must store stable provider or reference identifiers where available to preserve continuity across symbol, name, or identifier changes. Candidate listings are not app-visible. They feed primary/best-listing selection and later validation.

## Expected Output
Discovered candidate listings and identifier records can be stored and updated idempotently.

## Acceptance Criteria
- Candidate listings are unique by an explicit stable key appropriate to the discovered data.
- Candidate listings are linked to exchange and instrument type.
- Identifier records can hold multiple identifier types per candidate/listing without duplicates.
- Candidate upsert preserves continuity when symbol or name changes but a stable identifier remains available.
- Candidate records remain internal and are not returned by public app universe retrieval.

## Test Requirements
- Migration/schema tests for uniqueness and relationships.
- Upsert tests for new candidate, repeated candidate, and stable-identifier symbol/name change.
- Tests proving candidates are internal-only.

## Dependencies
- T1.
- T4.

## Stop Condition
Stop when normalized discovery output can be persisted as internal candidate and identifier records.
