# T6: Supported Universe State Persistence

## Purpose
Represent the current support state for chosen listings independently from raw discovery candidates.

## Scope
- Create or extend supported universe state records for selected listings.
- Track support status needed by Milestone 2, including discovered candidate, supported for price data, excluded, degraded, restoration-ready/restored where applicable, delisting-suspected, removed, and historical-only state if the existing enum pattern supports explicit states.
- Store state reason, effective day, benchmark-protection flag, signal-readiness placeholder, and app-visible eligibility.

## Allowed Changes
- Supported universe state model/module.
- Database migrations for support-state fields and indexes.
- Domain helpers for reading app-visible supported states.
- Focused state persistence tests.

## Forbidden Changes
- Do not implement ranking or lifecycle transition rules in this task.
- Do not implement price ingestion, signal calculation, or publication readiness.
- Do not expose excluded candidates through public APIs.
- Do not add broad operator dashboards.

## Inputs
- T5 candidate listing and identifier persistence.
- Milestone 1 core domain schema if it already contains instrument/listing foundations.

## Relevant Context
The app must see only the final supported universe, not unsupported candidates. The system needs enough support state to feed ingestion and later publication denominators. Signal generation may be unavailable until enough history exists, so signal-readiness must be distinguishable from price support.

## Expected Output
The database can distinguish internal candidates from app-visible supported universe members and store support-related state fields.

## Acceptance Criteria
- Supported universe records link to chosen listing/candidate identity.
- App-visible eligibility can be derived from support state.
- Excluded, degraded, delisting-suspected, restored, removed, and historical-only concepts can be represented without inventing publication behavior.
- Signal-readiness placeholder can represent not-ready without blocking price support.
- Indexes support querying supported instruments by exchange and instrument type.

## Test Requirements
- Schema/relationship tests for supported universe records.
- Unit tests for app-visible eligibility filtering.
- Unit tests for signal-readiness placeholder behavior.

## Dependencies
- T5.

## Stop Condition
Stop when supported universe state can be persisted and queried separately from discovery candidates.
