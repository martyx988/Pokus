## T6: Global Primary/Best-Listing Ranking Service

### Purpose
Implement deterministic listing ranking policy: home exchange, then turnover, then exchange activity priority.

### Scope
Build ranking service logic and decision evidence output for selected listing choice.

### Allowed Changes
- Ranking policy service/module
- Ranking input/read helper modules
- Decision-evidence data structure updates tied to ranking output
- Focused unit tests for ranking precedence

### Forbidden Changes
- Do not expose app APIs.
- Do not implement support-state transitions beyond chosen listing selection output.
- Do not implement exclusion/degradation lifecycle rules.

### Inputs
- Candidate listings from T5
- `design_phase/product_spec.md` FR-6 and AC-4
- Existing exchange metadata from T2

### Relevant Context
M2 acceptance requires primary/best-listing decisions explainable by stored ranking inputs.

### Implementation Budget
- Production files: max 2-4 ranking-focused files.
- Behavior count: 1 behavior (deterministic listing selection).
- Model/API/workflow count: 0-1 evidence-field model touch, 0 APIs, 1 ranking behavior.
- Test scope: precedence correctness and explainability payload completeness.

### Expected Output
Ranking service selects one listing per multi-listed entity with reproducible evidence of ordering inputs.

### Acceptance Criteria
- Ranking order is exactly home exchange > turnover > exchange activity priority.
- Ranking output includes decision evidence used for selection.
- Tie scenarios resolve deterministically.

### Test Requirements
- Unit tests for each precedence layer and tie-break path.

### Dependencies
- T5.

### Stop Condition
Stop when ranking decisions are deterministic and evidence-bearing.
