# T7: Universe Change Audit Records

## Purpose
Record lightweight audit history for supported-universe additions, exclusions, degradations, restorations, symbol/name/identifier changes, and removals.

## Scope
- Persist universe-change records linked to instrument/listing, exchange, instrument type, old state, new state, effective day, event type, reason, details, and actor/source where available.
- Support event types required for Milestone 2 and later operator filtering.
- Provide a small service/helper to write change records consistently from universe workflows.

## Allowed Changes
- Universe-change record model/module.
- Database migrations for universe-change history.
- Audit helper/service used by later tasks.
- Focused audit persistence tests.

## Forbidden Changes
- Do not implement operator dashboard screens in this task.
- Do not implement lifecycle decision rules.
- Do not create price/load/publication records.
- Do not expose unsupported candidate details through public APIs.

## Inputs
- T5 candidate/listing identity.
- T6 supported universe state.

## Relevant Context
The product requires auditable universe-change history. Required event concepts include additions, removals, exclusions, delisting suspicion, restored state, degraded state, degradation cleared, and symbol/name/identifier changes. Rows must preserve reason, details, and old/new state for investigation.

## Expected Output
Universe state changes can be recorded as durable, queryable audit records.

## Acceptance Criteria
- Change records can represent `added`, `removed`, `excluded`, `delisting_suspected`, `delisted_removed`, `restored`, `degraded`, `degradation_cleared`, `symbol_changed`, `name_changed`, and `identifier_changed`.
- Records include effective day, event type, instrument/listing reference, exchange, instrument type, reason, details, old state, and new state.
- Duplicate change records are avoided for idempotent workflow retries.
- Change records are internal/operator-facing only.

## Test Requirements
- Migration/schema tests for required fields and indexes.
- Unit tests for idempotent audit helper behavior.
- Unit tests for old/new state and reason validation.

## Dependencies
- T5.
- T6.

## Stop Condition
Stop when universe-change audit records can be written consistently and idempotently.
