# T1: Launch Scope Configuration Persistence

## Purpose
Create persistent admin-configurable scope records for supported exchanges and supported instrument types.

## Scope
- Represent launch exchanges: `NYSE`, `NASDAQ`, and `PSE`.
- Represent launch instrument types: stocks, ETFs, and ETNs.
- Persist whether each exchange and instrument type is enabled for support.
- Include stable display names, local timezone/calendar identifiers where available, currency metadata where available, and audit timestamps.

## Allowed Changes
- Domain/configuration models for exchanges and instrument types.
- Database migrations for exchange and instrument-type configuration.
- Seed/reference-data loading for launch exchanges and instrument types.
- Focused model and migration tests.

## Forbidden Changes
- Do not implement provider discovery, price ingestion, publication, or signal behavior.
- Do not add crypto support beyond preserving existing deferred `CRY` compatibility if already present.
- Do not hardcode app-visible universe results.
- Do not change unrelated Milestone 1 job, health, auth, or logging behavior except where required to reference the new config records.

## Inputs
- Milestone 1 domain and admin foundations.
- `task_phase/roadmap_plan.md` Milestone 2 scope.
- `design_phase/product_spec.md` launch scope and exchange/universe requirements.
- `design_phase/architecture.md` PostgreSQL/domain model direction.

## Relevant Context
The launch production scope is NYSE, Nasdaq, and Prague Stock Exchange. Launch instrument types are stocks, ETFs, and ETNs. Admin configuration must define supported exchanges and instrument types so the worker can discover the intended universe rather than relying on hidden constants.

## Expected Output
Persistent exchange and instrument-type configuration records exist and can be seeded for all launch scope values.

## Acceptance Criteria
- The database can store `NYSE`, `NASDAQ`, and `PSE` as supported exchange candidates.
- The database can store stocks, ETFs, and ETNs as supported instrument types.
- Scope records can be enabled or disabled without code changes.
- Re-running seed/reference-data loading is idempotent.
- Records expose enough metadata for later calendar, discovery, and app-visible universe work.

## Test Requirements
- Migration test or equivalent schema validation for required fields and uniqueness.
- Idempotent seed/reference-data test.
- Unit tests for enabled/disabled scope filtering.

## Dependencies
- Milestone 1 foundations only.

## Stop Condition
Stop when launch exchange and instrument-type scope can be represented, persisted, and loaded idempotently.
