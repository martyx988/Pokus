# T11: Initial Universe Discovery Workflow

## Purpose
Run an end-to-end internal workflow that discovers candidates for configured scope and produces a supported universe.

## Scope
- Read enabled exchange and instrument-type scope.
- Call the discovery adapter contract for each configured exchange/type pair.
- Persist candidate listings and identifiers.
- Recompute exchange activity priority when discovery provides enough turnover data.
- Apply primary/best-listing selection.
- Apply basic lifecycle helpers to create supported states and universe-change records.
- Treat market-closed days correctly when the workflow is date-sensitive.

## Allowed Changes
- Worker/service workflow for universe discovery.
- Job registration or command entry point under the existing Milestone 1 worker/job system.
- Integration tests using fake/fixture discovery adapters.
- Structured logs for discovery workflow start, finish, failures, and universe changes.

## Forbidden Changes
- Do not implement real provider validation, provider reliability scoring, price ingestion, publication, or signal generation.
- Do not expose unsupported candidates through public APIs.
- Do not add scheduler complexity beyond a minimal runnable job/command if the existing job framework supports it.
- Do not silently ignore adapter failures.

## Inputs
- T1 admin scope records.
- T3 calendar decisions.
- T4 discovery adapter contract.
- T5 candidate and identifier persistence.
- T8 exchange activity priority.
- T9 best-listing policy.
- T10 lifecycle helpers.

## Relevant Context
Milestone 2 must prove that configured launch scope can produce an eligible supported universe and auditable universe-change history. The workflow feeds later ingestion by determining eligible instruments. Provider validation and price loading remain later milestones.

## Expected Output
A worker-executable discovery workflow can populate candidate listings, supported universe state, and universe-change history from deterministic discovery input.

## Acceptance Criteria
- The workflow processes enabled launch exchanges and instrument types only.
- Candidate records are idempotently upserted.
- Supported universe state is created or updated for selected listings.
- Additions, exclusions, restorations, degradations, symbol/name/identifier changes, and removals are recorded when the input changes require them.
- Adapter failures leave a visible job/log outcome and do not create partial unsupported app visibility.
- Re-running the workflow with the same fake input does not duplicate candidates or audit records.

## Test Requirements
- Integration test for first discovery run.
- Integration test for idempotent rerun.
- Integration test for changed symbol/name/identifier input.
- Failure-path test for adapter error handling.
- Test that disabled scope is skipped.

## Dependencies
- T1 through T10.

## Stop Condition
Stop when the internal discovery workflow can produce a supported universe and audit records from fixture adapter data.
