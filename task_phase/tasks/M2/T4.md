# T4: Instrument Discovery Adapter Contract

## Purpose
Define the provider-agnostic contract for instrument discovery without implementing provider validation or price loading.

## Scope
- Define normalized discovery candidate fields for exchange, instrument type, symbol, name, currency, provider/reference identifiers, home exchange indicator where available, and turnover/traded-value inputs where available.
- Define adapter result and error shapes for discovery runs.
- Provide a minimal fake or fixture adapter for integration tests.

## Allowed Changes
- Provider adapter interface modules related to discovery only.
- Discovery DTOs, schemas, or typed records.
- Fake/fixture adapter code used only by tests or local workflow validation.
- Contract tests for normalized discovery candidates.

## Forbidden Changes
- Do not implement real provider integrations, source prioritization, provider reliability scoring, or price candidate capture.
- Do not store price values.
- Do not choose supported listings in this task.
- Do not expose discovery candidates to public app APIs.

## Inputs
- T1 configured launch scope.
- Existing provider adapter boundary from Milestone 1 if present.

## Relevant Context
The architecture isolates free market-data providers behind adapters. Milestone 2 needs the discovery boundary and initial workflow; Milestone 3 handles provider validation, provider attempts, source prioritization, and reliability scoring. Candidate data must be normalized before universe selection.

## Expected Output
A stable discovery adapter contract exists and can return normalized candidate listing records for configured exchange/type scope.

## Acceptance Criteria
- Discovery candidates include enough fields to feed candidate listing storage and best-listing selection.
- Adapter failures are represented in a controlled way for worker handling.
- A fake or fixture adapter can produce deterministic candidates in tests.
- Contract tests verify required normalized fields and invalid candidate handling.

## Test Requirements
- Unit/contract tests for valid normalized candidates.
- Tests for missing required candidate fields.
- Tests for adapter failure result handling.

## Dependencies
- T1.

## Stop Condition
Stop when discovery adapters have a typed/tested boundary but no real provider validation or price behavior has been added.
