# T10: Universe Quality and Lifecycle Transition Helpers

## Purpose
Provide focused domain helpers for support-state transitions and exclusion/degradation reasons used by universe maintenance.

## Scope
- Implement transition helpers for additions, exclusions, degradation, degradation cleared, restoration, delisting suspicion, delisted removal, symbol/name/identifier changes, and historical-only state changes where represented.
- Enforce reason codes for low turnover, stale/missing data, benchmark protection, degradation, restoration, delisting suspicion, symbol change, name change, identifier change, and removal.
- Write universe-change records through the T7 audit helper.

## Allowed Changes
- Supported universe lifecycle service/module.
- Reason-code enum/constants owned by universe management.
- Tests for transition validation and audit writes.

## Forbidden Changes
- Do not implement price-load outcome scanning or publication denominator rules.
- Do not calculate stale/missing counts from price records.
- Do not implement operator override workflows beyond accepting an actor/reason if the existing foundation already supports it.
- Do not create broad dashboards.

## Inputs
- T6 supported universe state.
- T7 universe-change audit helper.
- T3 calendar service only if needed for delisting-suspicion effective-day tests.

## Relevant Context
Milestone 2 must include basic quality and exclusion state fields for low turnover, stale/missing data, delisting suspicion, benchmark protection, degradation, restoration, and historical-only status. The full data-driven triggers depend on later price/load milestones, but state transitions and audit recording must exist now.

## Expected Output
Universe support state can be changed through validated helpers that consistently record audit history.

## Acceptance Criteria
- Each supported lifecycle event updates state consistently.
- Each state change writes an appropriate universe-change audit record.
- Invalid transitions are rejected with clear errors.
- Benchmark-protected records cannot be automatically excluded without an explicit protected reason path.
- Symbol, name, and identifier changes create audit records without requiring a full corporate-action engine.

## Test Requirements
- Unit tests for each transition type.
- Invalid transition tests.
- Audit record tests for old/new state, reason, and details.
- Benchmark-protection behavior test.

## Dependencies
- T6.
- T7.

## Stop Condition
Stop when lifecycle state changes are centralized, validated, and auditable.
