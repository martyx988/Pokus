## T002: API and Worker Runtime Roles

### Purpose
Define how the API/web process and worker/scheduler process run independently against the same PostgreSQL database.

### Scope
Add Docker Compose services, runtime entrypoints, and settings required to start the API role, worker role, scheduler role if separate, and PostgreSQL.

### Allowed Changes
- `project/docker-compose.yml`
- `project/Dockerfile`
- `project/docker/`
- `project/config/settings/`
- `project/config/wsgi.py`
- `project/config/asgi.py`
- `project/manage.py`
- `project/README.md`

### Forbidden Changes
- Do not introduce Redis, RabbitMQ, Kafka, or a managed database dependency.
- Do not implement real jobs, ingestion, provider adapters, or publication logic.
- Do not expose private operator/admin endpoints beyond framework defaults.
- Do not modify task or design documents.

### Inputs
- T001 project skeleton
- Environment variable names from `.env.example`

### Relevant Context
The selected architecture deploys one codebase as two process types: API/web serves mobile and private operator/admin surfaces, while worker/scheduler owns scheduled and asynchronous backend work. Deployment is VPS-first using Docker Compose with PostgreSQL.

### Expected Output
Compose and entrypoint configuration can start API/web and worker/scheduler roles as separate services sharing one PostgreSQL database.

### Acceptance Criteria
- `docker compose config` succeeds.
- API and worker services use separate commands.
- Both roles receive the same database configuration.
- The default configuration does not require Redis.
- Documentation identifies the commands used to run API and worker locally.

### Test Requirements
- Validate Compose configuration.
- Add or update a smoke test that settings can load for API and worker mode.

### Dependencies
T001.

### Stop Condition
Stop when the two process roles are defined and startable without adding business behavior.
