## T013: Basic Public and Private Access Boundaries

### Purpose
Establish minimal authentication and permission boundaries for public app reads and private operator/admin surfaces.

### Scope
Create DRF permission classes, middleware or authentication hooks as needed, and smoke-test placeholder routes only if required to prove boundaries.

### Allowed Changes
- `project/apps/api/`
- `project/apps/operations/`
- `project/config/urls.py`
- `project/config/settings/`
- `project/apps/api/tests/`
- `project/apps/operations/tests/`

### Forbidden Changes
- Do not implement real market-data endpoints.
- Do not expose unsupported instruments, prices, signals, or operator dashboards.
- Do not store secrets in source control.
- Do not add heavy user-management workflows beyond Django's standard authentication support.

### Inputs
- T001 project skeleton
- T011 admin command foundation

### Relevant Context
Mobile API credentials are app identification and throttling controls, not strong user authentication. Operator/admin access requires authenticated sessions and should remain private by deployment path where practical. Public clients can read only supported, app-visible, published data in later milestones.

### Expected Output
The codebase contains reusable permission/auth boundary components for future public app endpoints and private operator/admin endpoints.

### Acceptance Criteria
- Public app permission checks require configured app identification or signed-token placeholder behavior.
- Private operator/admin permission checks require authenticated staff/operator access.
- Tests prove unauthenticated requests cannot access private placeholder routes.
- No real secrets are committed.

### Test Requirements
- DRF permission tests for public app identification accepted/rejected cases.
- DRF permission tests for private operator/admin authenticated/unauthenticated cases.

### Dependencies
T001, T011.

### Stop Condition
Stop when access-boundary components are tested without real product endpoints.
