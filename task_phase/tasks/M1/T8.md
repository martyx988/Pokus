## T8: Load Job Model and State Transition Service

### Purpose
Implement durable job records and strict job-state transition semantics.

### Scope
Add `LoadJob` schema and a small domain service for allowed state transitions, terminal-state detection, idempotency keys, locks, retry metadata, and worker heartbeat/lock-expiry fields.

### Allowed Changes
- Domain model files for jobs
- Job state transition service/module
- Migrations for job tables
- Focused job tests

### Forbidden Changes
- Do not implement provider ingestion, scheduling, or worker task execution.
- Do not add manual retry/cancel admin APIs.
- Do not implement publication readiness calculation.

### Inputs
- T2 database baseline
- Product job states: `queued`, `running`, `retry_wait`, `succeeded`, `failed`, `cancelled`, `stale_abandoned`

### Relevant Context
Jobs must persist state transitions. Only `succeeded`, `failed`, and `cancelled` count as terminal outcomes for publication-readiness logic. Idempotency keys, locks, bounded retry metadata, request timeout metadata, heartbeat, and lock expiry are required so crashed work can be recovered later.

### Implementation Budget
- Production files: expected maximum 2-4 closely related model/service/migration files.
- Behavior count: 1 behavior, valid job state transition and terminal-state semantics.
- Model/API/workflow count: 1 model entity, 0 APIs, 1 domain service.
- Test scope: allowed/forbidden transitions, terminal-state semantics, idempotency uniqueness, lock/heartbeat field behavior.

### Expected Output
`LoadJob` exists with a tested transition service and durable fields for later worker execution.

### Acceptance Criteria
- All required job states exist.
- Terminal states are exactly `succeeded`, `failed`, and `cancelled`.
- Invalid transitions are rejected or fail clearly.
- Idempotency keys prevent duplicate active work units.
- Retry, lock, heartbeat, and stale-abandoned support fields exist without implementing recovery scheduling.

### Test Requirements
Test every allowed transition, representative forbidden transitions, terminal-state helper behavior, and idempotency-key uniqueness.

### Dependencies
T2.

### Stop Condition
Stop when job records and transition semantics are implemented and tested without running jobs.
