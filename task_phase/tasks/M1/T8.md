## T8: Add Exchange Calendar Abstraction

### Purpose
Create the exchange-calendar boundary needed for later exchange/day lifecycle decisions without hardcoding business logic into ingestion or publication code.

### Scope
- Add a calendar service interface for checking whether an exchange is open on a local trading date.
- Add a provider implementation using `pandas-market-calendars` where available.
- Add a manual override model or configuration hook for holidays, special closures, late opens, and source gaps.
- Add tests using fixed dates and controlled overrides.

### Allowed Changes
- `project/apps/domain/models.py` or calendar-specific model files for override storage.
- `project/apps/domain/migrations/**`
- `project/apps/domain/services/calendars.py`
- `project/apps/domain/tests/test_calendars*.py`
- Dependency metadata if `pandas-market-calendars` is added.

### Forbidden Changes
- Do not implement exchange onboarding validation.
- Do not create custom calendars unless needed as a simple adapter stub.
- Do not schedule ingestion based on calendar results.
- Do not decide publication readiness.
- Do not add unsupported exchanges beyond test fixtures.

### Inputs
- T3 exchange schema.
- `design_phase/architecture.md` Exchange Calendars and Data Ingestion Flow.
- `design_phase/product_spec.md` exchange-day lifecycle and launch constraints.

### Relevant Context
Calendar mistakes can cause false market-open or market-closed decisions. The architecture requires an exchange-agnostic calendar validation flow, with manual override records for known official-calendar mismatches.

### Expected Output
- A calendar service can answer open/closed status for an exchange/date.
- Manual override records can supersede library results.
- Unsupported or unmapped exchanges fail clearly.

### Acceptance Criteria
- Tests verify library-backed open/closed checks for at least one known mapped exchange using fixed dates.
- Tests verify a manual closed-day override changes the calendar result.
- Tests verify a manual open-day override changes the calendar result.
- Tests verify unknown exchange-calendar mapping returns a clear error.

### Test Requirements
- Unit tests for service contract.
- Unit tests for override precedence.
- Model tests for calendar override validation.

### Dependencies
T3.

### Stop Condition
Stop when calendar queries are available through one tested service boundary and no ingestion code depends on calendar internals.
