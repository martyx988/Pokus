## T012: Exchange Calendar Abstraction

### Purpose
Introduce the calendar interface needed to decide expected trading days and market-closed states later.

### Scope
Create a small exchange-calendar service abstraction, a deterministic test adapter, and unit tests for weekend/holiday/closed-day behavior.

### Allowed Changes
- `project/apps/calendar/`
- `project/apps/calendar/tests/`
- `project/config/settings/` for app registration only if needed

### Forbidden Changes
- Do not implement full NYSE, Nasdaq, or Prague calendar adapters.
- Do not schedule jobs or mark exchange-day loads market closed.
- Do not call external calendar APIs.
- Do not change domain models unless a lightweight interface import is required.

### Inputs
- T003 core identity models

### Relevant Context
Calendar mistakes can cause false market-open or market-closed decisions. Later workflows depend on expected trading days for load scheduling, delisting-suspicion counts, and exchange/day publication behavior.

### Expected Output
A tested calendar interface exists with a fake/in-memory implementation that future exchange-specific adapters can implement.

### Acceptance Criteria
- Calendar service exposes methods to check whether an exchange/date is an expected trading day and to explain closed days.
- Tests cover weekends, configured holidays, and open weekdays using the deterministic adapter.
- The abstraction accepts an exchange identifier without hardcoding future launch-exchange behavior.

### Test Requirements
- Unit tests for expected trading day decisions.
- Unit tests for closed-day reason output.

### Dependencies
T003.

### Stop Condition
Stop when the abstraction is available and tested without production exchange calendars.
