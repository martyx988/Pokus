## T5: Create Load Job, Outcome, Exchange-Day Load, Admin Command, and Audit Schema

### Purpose
Create durable schema for background job state, per-instrument load outcomes, exchange-day aggregate loads, admin commands, and auditable operational history.

### Scope
- Add `LoadJob`, `InstrumentLoadOutcome`, `ExchangeDayLoad`, and `AdminCommand` models.
- Add fields for job state, job type/load type, attempts, idempotency key, lock/heartbeat timestamps, failure reason, started/finished timestamps, and operator/admin reason fields.
- Add terminal outcome representation for per-instrument load results.
- Add audit-oriented fields needed for admin-command visibility and later operator dashboards.

### Allowed Changes
- `project/apps/domain/models.py` or split files under `project/apps/domain/models/**`
- `project/apps/domain/migrations/**`
- `project/apps/domain/constants.py`
- Domain tests under `project/apps/domain/tests/**`
- Basic admin registration for operational models.

### Forbidden Changes
- Do not implement job execution.
- Do not implement retries, cancellation handlers, ingestion, validation, or publication logic.
- Do not create operator dashboards.
- Do not expose admin commands through public APIs.
- Do not create external queue infrastructure.

### Inputs
- T3 and T4 schema.
- `design_phase/architecture.md` Worker/Scheduler Process, Data Model Overview, Job Failures, and Operator/Admin Module.
- `task_phase/roadmap_plan.md` Milestone 1 job state acceptance criteria.

### Relevant Context
Allowed job states are `queued`, `running`, `retry_wait`, `succeeded`, `failed`, `cancelled`, and `stale_abandoned`. Only `succeeded`, `failed`, and `cancelled` are terminal for publication-readiness logic. Jobs must later be idempotent by exchange/day/instrument/load type.

### Expected Output
- Durable job and load outcome tables exist.
- Job-state constants are centralized for later services.
- Schema stores enough metadata for job lifecycle logs, retries, lock expiry, manual admin actions, and publication readiness.

### Acceptance Criteria
- Migrations apply cleanly after T4.
- Tests can create each allowed job state.
- Tests verify the terminal publication states are exactly `succeeded`, `failed`, and `cancelled`.
- Idempotency key fields and active-job uniqueness constraints exist for the intended unit of work.

### Test Requirements
- Model tests for allowed job states.
- Constraint tests for duplicate active jobs with the same idempotency key.
- Model tests for exchange-day load status values including not started, in progress, ready, partial/problematic, failed, and market closed.

### Dependencies
T4.

### Stop Condition
Stop when durable operational schema supports job state and exchange/day lifecycle representation without running jobs.
