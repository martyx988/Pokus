## T7: Add Worker Scheduler Shell and Heartbeat Persistence

### Purpose
Create the minimal worker/scheduler runtime shell that records liveness and queue metadata without performing real market-data work.

### Scope
- Add a worker management command or process entrypoint.
- Add scheduler heartbeat persistence using the operational schema.
- Add worker heartbeat persistence.
- Add queue depth or pending-job-age calculation from database-backed jobs.
- Integrate Django-Q2 only as a PostgreSQL ORM-broker configuration if the project chooses to wire the library in this milestone.

### Allowed Changes
- `project/apps/worker/**`
- `project/apps/operations/**`
- `project/config/settings/**`
- Tests under `project/apps/worker/tests/**` and `project/apps/operations/tests/**`
- Minimal `LoadJob` selector additions needed to compute queue age/depth.

### Forbidden Changes
- Do not implement scheduled ingestion, validation, universe discovery, publication, signal generation, or historical fill.
- Do not add Redis or any non-PostgreSQL broker.
- Do not run long-lived background loops in test code.
- Do not mutate job states except heartbeat/queue inspection unless directly using the T6 service.

### Inputs
- T6 job state service.
- `design_phase/architecture.md` Background Jobs / Queue, Worker/Scheduler Process, Health Checks, and Deployment / Hosting.
- `task_phase/roadmap_plan.md` Milestone 1 health and worker acceptance criteria.

### Relevant Context
Milestone 1 must prove API and worker roles can share durable state and expose worker/scheduler liveness. Real provider ingestion belongs to future milestones.

### Expected Output
- Worker startup can record a heartbeat.
- Scheduler heartbeat can be updated by a minimal command/function.
- Queue depth or oldest pending job age can be computed from database jobs.

### Acceptance Criteria
- Worker heartbeat records are visible in the database.
- Scheduler heartbeat records are visible in the database.
- Queue depth or pending job age returns deterministic values for queued/retry_wait jobs.
- Worker startup does not require provider credentials or external data access.

### Test Requirements
- Unit or integration test for worker heartbeat write.
- Unit or integration test for scheduler heartbeat write.
- Unit test for queue depth or pending-job-age selector.
- Smoke test for the worker entrypoint.

### Dependencies
T6.

### Stop Condition
Stop when worker and scheduler liveness can be persisted and queried with no market-data side effects.
