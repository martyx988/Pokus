## T11: Add Structured Logging and Metrics Plumbing

### Purpose
Create the initial observability plumbing for job lifecycle events, admin-command events, and core operational metrics.

### Scope
- Configure structured application logging.
- Add helper functions or adapters for job lifecycle log events.
- Add helper functions or adapters for admin-command log events.
- Add basic metrics collection/export primitives for exchange/day status, coverage, load counts, job retry counts, API latency, and error rate where placeholders are needed.
- Ensure sensitive values are not logged.

### Allowed Changes
- `project/config/settings/**`
- `project/apps/operations/logging.py`
- `project/apps/operations/metrics.py`
- `project/apps/domain/services/jobs.py` only to emit job lifecycle logs through the new helper.
- Tests under `project/apps/operations/tests/**` and existing job tests if log assertions are added.

### Forbidden Changes
- Do not add a hosted observability vendor.
- Do not implement alert routing.
- Do not log provider keys, app credentials, admin credentials, signing secrets, or database passwords.
- Do not implement provider attempt logging beyond helper contracts unless provider attempt schema already exists.
- Do not change job-state behavior except to emit structured events.

### Inputs
- T6 job state service.
- T9 authentication boundary.
- `design_phase/architecture.md` Observability, Logging, Metrics, Alerts, and Sensitive Data Handling.
- `task_phase/roadmap_plan.md` Milestone 1 structured logging acceptance criteria.

### Relevant Context
The architecture requires structured logs for job lifecycle and admin-command events from Milestone 1, with broader provider, quality, publication, signal, and universe logs added as those workflows are implemented. Metrics begin simple and owner-focused.

### Expected Output
- Logs are structured consistently.
- Job lifecycle transitions emit structured events.
- Admin-command logging helper exists for later admin actions.
- Metrics helpers are available without requiring final dashboards.

### Acceptance Criteria
- Tests verify job start/finish/failure-style events are emitted with job id, job type, state, and idempotency key.
- Tests verify admin-command event helper emits command type, actor id or system marker, status, and reason without secrets.
- Tests verify configured logging output is structured.
- Tests verify metrics helpers can record or expose the required initial metric names.

### Test Requirements
- Unit tests using log capture.
- Unit tests for secret redaction or explicit non-inclusion.
- Unit tests for metrics helper registration or serialization.

### Dependencies
T6, T9.

### Stop Condition
Stop when implementation agents can emit structured operational logs and metrics through shared helpers rather than ad hoc logging.
