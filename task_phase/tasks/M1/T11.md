## T11: Signal Statistics and Signal Event Schema

### Purpose
Reserve the durable schema for backend-owned `Dip` and `Skyrocket` signal outputs and supporting statistics.

### Scope
Add `SignalStatistic` and `SignalEvent` models/migrations with fields for algorithm version/context, signal type, trading date, instrument/listing references, and insufficient-history/no-signal representation as durable state where needed.

### Allowed Changes
- Domain model files for signal statistics and events
- Migrations for signal tables
- Focused model tests

### Forbidden Changes
- Do not implement the existing alert algorithm adapter.
- Do not compute historical or daily signals.
- Do not implement app signal endpoints or retrospective recomputation behavior.

### Inputs
- T4 instrument/listing schema
- T7 price record schema
- Product rule that backend computes only `Dip` and `Skyrocket`

### Relevant Context
The backend generates only `Dip` and `Skyrocket` using an existing algorithm. It must persist supporting statistics and reasoning context, distinguish no signal from insufficient-history unavailability, and avoid retrospective post-launch rewrites after algorithm changes.

### Implementation Budget
- Production files: expected maximum 1-3 closely related domain/migration files.
- Behavior count: 1 behavior, durable signal/statistic representation.
- Model/API/workflow count: 2 model entities, 0 APIs, 0 workflows.
- Test scope: signal type constraints, uniqueness, relationships to listing/date/algorithm version, and statistics context storage.

### Expected Output
Signal statistics and events can be stored for later initial fill and daily signal generation workflows.

### Acceptance Criteria
- Only `Dip` and `Skyrocket` event types are valid backend signal events.
- Supporting statistics can be associated with an instrument/listing and trading date.
- Algorithm version or equivalent provenance is representable.
- The schema can distinguish signal event, no signal, and signal unavailable when later services populate it.

### Test Requirements
Test signal type constraints, required provenance/context fields, and duplicate prevention for the same listing/date/signal type where appropriate.

### Dependencies
T4, T7.

### Stop Condition
Stop when signal schemas are migrated and tested without algorithm execution.

- Done (2026-05-01)

  - algorithm provenance (`algorithm_version`) and JSON context storage
