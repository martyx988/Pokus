## T4: Instrument Identity and Listing Schema

### Purpose
Create the schema for canonical instruments, exchange-specific listings, stable identifiers, and current support state.

### Scope
Add models and migrations for `Instrument`, `Listing`, `IdentifierRecord`, and `SupportedUniverseState`.

### Allowed Changes
- Domain model files for instruments and listings
- Migrations for instrument identity and support-state tables
- Focused model tests

### Forbidden Changes
- Do not implement discovery adapters or primary/best-listing selection logic.
- Do not implement universe-change audit records.
- Do not expose public supported-universe APIs.

### Inputs
- T3 exchange and instrument type schema
- Product rules for one chosen supported listing and stable identifier storage

### Relevant Context
The app must see only the final supported universe. A canonical instrument may have candidate listings, but one chosen supported listing is app-visible under later selection rules. Stable provider/reference identifiers are stored when available to preserve continuity across symbol or name changes.

### Implementation Budget
- Production files: expected maximum 1-3 closely related domain/migration files.
- Behavior count: 1 behavior, durable representation of instrument identity and listing support state.
- Model/API/workflow count: 4 model entities, 0 APIs, 0 workflows.
- Test scope: relationships, uniqueness constraints, and support-state representation.

### Expected Output
Instrument identity, listing, identifier, and support-state tables exist and can be related to exchanges and instrument types.

### Acceptance Criteria
- Instruments are separate from exchange-specific listings.
- Listings reference an exchange and instrument.
- Identifier records can store provider/reference identifiers without requiring a full corporate-action engine.
- Supported-universe state can represent supported, not-yet-signal-eligible, delisting-suspected, removed, and historical-only concepts as durable state values or fields.

### Test Requirements
Test model relationships, uniqueness boundaries, and valid support-state transitions only as representable data, not workflow behavior.

### Dependencies
T3.

### Stop Condition
Stop when the identity/listing schema is migrated and tested without discovery behavior.
