## T4: Create Provider, Price, Signal, Publication, Quality, and Read-Model Schema

### Purpose
Create the storage boundaries for provider attempts, trusted price records, signal outputs/statistics, publication state, quality checks, and app/operator read models.

### Scope
- Add schema entities for provider metadata and provider attempts.
- Add price records for adjusted historical close and official exchange-local unadjusted current-day open.
- Add signal statistics and `Dip`/`Skyrocket` signal events.
- Add publication, quality-check, and read-model placeholder tables needed by later milestones.

### Allowed Changes
- `project/apps/domain/models.py` or split files under `project/apps/domain/models/**`
- `project/apps/domain/migrations/**`
- Domain tests under `project/apps/domain/tests/**`
- Basic admin registration if already introduced in T3.

### Forbidden Changes
- Do not call external providers.
- Do not implement source prioritization, correctness validation, publication decisions, or signal computation.
- Do not create app-facing API serializers or views.
- Do not change the universe schema from T3 except where a direct foreign key is required.
- Do not persist user-specific alert rules.

### Inputs
- T3 universe schema.
- `design_phase/architecture.md` Data Model Overview, Publication Module, Read Model Builder, and Signal Generation Module.
- `design_phase/product_spec.md` sections 3.2, 3.3, 3.4, 5.4, 5.5, and acceptance criteria AC-6 through AC-14.

### Relevant Context
Price records must store currency and preserve the distinction between adjusted historical close and official unadjusted current-day open. Signal events are backend-generated `Dip` and `Skyrocket` only. Published data is stable unless explicit reprocessing later changes historical adjusted-close records.

### Expected Output
- Models exist for `Provider`, `ProviderAttempt`, `PriceRecord`, `SignalStatistic`, `SignalEvent`, `PublicationRecord`, `QualityCheck`, and read-model storage.
- Price uniqueness prevents more than one trusted price per instrument/listing, trading date, and price type after source prioritization.
- Publication records can represent exchange/day publication state and quality summary.

### Acceptance Criteria
- Migrations apply cleanly after T3.
- Tests can create historical close and current-day open price records with currencies.
- Tests verify only `Dip` and `Skyrocket` are accepted signal event types.
- Publication and quality records can be associated with an exchange/day.

### Test Requirements
- Model tests for price-type validation and currency requirement.
- Constraint tests for duplicate trusted price records.
- Model tests for allowed signal types.
- Migration test or documented migration command against PostgreSQL.

### Dependencies
T3.

### Stop Condition
Stop when provider, price, signal, publication, quality, and read-model schema exists and is test-covered without implementing business workflows.
