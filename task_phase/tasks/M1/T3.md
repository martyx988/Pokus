## T3: Create Exchange, Instrument, Listing, and Universe Schema

### Purpose
Create the foundational relational schema for supported exchanges, instrument types, instruments, listings, identifiers, supported universe state, and universe-change history.

### Scope
- Add Django models and initial migrations for core universe entities.
- Encode launch-compatible instrument types for stocks, ETFs, ETNs, and future crypto without enabling crypto behavior.
- Represent exchange/day and listing relationships needed by future publication and universe tasks.
- Add database constraints for stable identity and app-visible support state.

### Allowed Changes
- `project/apps/domain/models.py` or split files under `project/apps/domain/models/**`
- `project/apps/domain/migrations/**`
- `project/apps/domain/admin.py` only for basic model registration.
- Domain model tests under `project/apps/domain/tests/**`.

### Forbidden Changes
- Do not implement universe discovery.
- Do not implement primary/best-listing selection logic.
- Do not implement crypto admission behavior or the synthetic `CRY` exchange beyond schema compatibility if needed.
- Do not expose public API endpoints.
- Do not create provider ingestion code.

### Inputs
- T1 and T2.
- `design_phase/architecture.md` Data Model Overview.
- `design_phase/product_spec.md` sections 5.1, 5.2, 5.5, and launch scope.

### Relevant Context
The system must launch with NYSE, Nasdaq, and Prague Stock Exchange later, expose only supported instruments to the app, track one chosen supported listing per instrument, and preserve symbol/name/identifier continuity through records and change history.

### Expected Output
- Models exist for `Exchange`, `InstrumentType`, `Instrument`, `Listing`, `IdentifierRecord`, `SupportedUniverseState`, and `UniverseChangeRecord`.
- Migrations create the schema in PostgreSQL.
- Constraints prevent duplicate core identities where the spec requires uniqueness.

### Acceptance Criteria
- Migrations apply cleanly on an empty PostgreSQL database.
- Tests can create an exchange, instrument type, instrument, listing, identifier record, supported-universe state, and universe-change record.
- App-visible support state can distinguish at least price-supported from signal-eligible status.
- Universe-change records can store event type, effective day, reason, details, old state, and new state.

### Test Requirements
- Model constraint tests for duplicate exchange codes, duplicate instrument types, and duplicate listing identity within an exchange.
- Model tests for allowed universe-change event types needed by the product specification.
- Migration test or documented migration command against PostgreSQL.

### Dependencies
T1, T2.

### Stop Condition
Stop when the universe foundation can be migrated and exercised through model tests without implementing discovery or selection behavior.
