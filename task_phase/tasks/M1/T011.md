## T011: Admin Command and Audit Event Foundation

### Purpose
Create persistent storage for restricted admin commands and their audit trail.

### Scope
Implement `AdminCommand` plus any small audit event helper required for command lifecycle logging.

### Allowed Changes
- `project/apps/operations/`
- `project/apps/operations/migrations/`
- `project/apps/operations/tests/`
- `project/config/settings/` for app registration only
- `project/apps/domain/models/` only for foreign-key relationships to existing domain records

### Forbidden Changes
- Do not implement admin command execution.
- Do not add reprocessing, validation, or retention behavior.
- Do not expose command endpoints.
- Do not implement broad user-management workflows.

### Inputs
- T003 core identity models
- T006 load job model

### Relevant Context
Admin users can change exchange/instrument-type configuration, trigger validation, trigger historical adjusted-close reprocessing, and apply retention cutoff. Administrative actions must be audited and require private operator/admin access in later tasks.

### Expected Output
The database can record an admin command request, actor reference, command type, parameters, reason, status, related job, and audit timestamps.

### Acceptance Criteria
- Command types include placeholders for exchange validation, explicit reprocessing, retention cutoff, and manual job actions.
- Commands require a reason for manual operational actions.
- Command records never store secrets.
- Tests cover required reason and status transitions.

### Test Requirements
- Model tests for command type/status choices.
- Model tests requiring reason for manual retry/cancel/mark-failed command types.

### Dependencies
T003, T006.

### Stop Condition
Stop when admin commands can be recorded and audited without execution behavior.
