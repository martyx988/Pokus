## T2: PostgreSQL Configuration and Migration Baseline

### Purpose
Establish PostgreSQL as the shared durable store for the API and worker roles.

### Scope
Configure database settings, migration tooling, local database connection behavior, and the first empty or framework baseline migration path.

### Allowed Changes
- `project/` database settings and migration configuration
- Local environment example values for PostgreSQL connection settings
- Database connectivity test files

### Forbidden Changes
- Do not add domain tables beyond framework-required baseline tables.
- Do not add SQLite as the production or default shared runtime database.
- Do not implement backup automation, read models, or queue semantics.

### Inputs
- T1 project skeleton
- Architecture decision to use PostgreSQL inside the VPS/Docker Compose deployment

### Relevant Context
PostgreSQL is the source of truth for domain data, historical data, job state, publication state, read models, admin commands, and audit records. API and worker processes must share the same database.

### Implementation Budget
- Production files: expected maximum 3-5 settings/config/migration-support files.
- Behavior count: 1 behavior, successful PostgreSQL connectivity through shared settings.
- Model/API/workflow count: 0 domain models, 0 APIs, 1 persistence boundary.
- Test scope: database connection and migration-command smoke coverage.

### Expected Output
The project can connect to PostgreSQL and run baseline migrations from both runtime roles.

### Acceptance Criteria
- Database settings are environment-driven and do not hardcode secrets.
- API and worker roles use the same PostgreSQL configuration.
- Baseline migrations can be applied on a clean database.
- Tests or checks fail clearly when PostgreSQL is unavailable.

### Test Requirements
Run migration and database connectivity checks using the project test command or documented local command.

### Dependencies
T1.

### Stop Condition
Stop when PostgreSQL connectivity and migration plumbing are in place without domain schema work.

### Status
Done.

### Completion Summary
Added shared PostgreSQL connectivity utilities and Alembic baseline migration plumbing in `Project/`, wired API and worker startup checks to the shared database settings, added local `.env.example` PostgreSQL values, and extended smoke tests to verify clear failures for unavailable PostgreSQL and migration execution paths.
