## T008: Publication and Quality Check Models

### Purpose
Create storage for exchange/day publication records and quality-check evidence used to gate app visibility.

### Scope
Implement model fields, constraints, migrations, admin registration, and tests for:
- `QualityCheck`
- `PublicationRecord`

### Allowed Changes
- `project/apps/domain/models/`
- `project/apps/domain/admin.py`
- `project/apps/domain/migrations/`
- `project/apps/domain/tests/`

### Forbidden Changes
- Do not implement benchmark validation algorithms.
- Do not mark real exchange/day records ready.
- Do not build app read models.
- Do not add public API endpoints.

### Inputs
- T007 exchange-day load and instrument outcome models

### Relevant Context
Current-day opening-price publication must remain unpublished until all eligible instruments have terminal outcomes, coverage exceeds 99%, and correctness validation succeeds. Publication records must preserve the load and quality state used for readiness.

### Expected Output
The schema can record quality-check results and publication decisions separately from load execution.

### Acceptance Criteria
- `QualityCheck` can store check type, status, mismatch percentage, coverage, summary details, and blocking reason.
- `PublicationRecord` references exchange/day/load type and the quality/load state it was based on.
- Publication status can represent unpublished, ready, partial/problematic, failed, and market closed.
- Ready publication records require an associated successful quality state through domain validation or tests.
- Tests cover status choices and ready-publication guardrails.

### Test Requirements
- Unit/model tests for allowed quality statuses.
- Unit/model tests preventing ready publication without successful quality evidence.

### Dependencies
T007.

### Stop Condition
Stop when publication and quality evidence can be stored without implementing publication computation.
