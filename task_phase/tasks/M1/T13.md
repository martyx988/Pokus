## T13: Exchange Calendar Abstraction

### Purpose
Create the internal calendar interface used to distinguish expected trading days from closed days.

### Scope
Add a calendar service abstraction, result types, and a simple provider boundary that can later wrap `pandas-market-calendars` and validation overrides.

### Allowed Changes
- Calendar service/interface modules
- Calendar result/value object modules
- Focused unit tests

### Forbidden Changes
- Do not implement full NYSE, Nasdaq, PSE validation datasets.
- Do not create custom calendar adapters unless required by later validation.
- Do not wire calendar decisions into ingestion or publication workflows.

### Inputs
- T3 exchange schema
- Architecture recommendation for exchange-agnostic calendar validation using `pandas-market-calendars` first

### Relevant Context
Calendar mistakes can cause false failures, missed loads, and incorrect delisting suspicion. The system must know market-open versus market-closed days before ingestion and publication workflows are built.

### Implementation Budget
- Production files: expected maximum 2-4 closely related calendar modules.
- Behavior count: 1 behavior, answer expected trading day versus market closed through an abstraction.
- Model/API/workflow count: 0 models, 0 APIs, 1 service boundary.
- Test scope: interface behavior with stubbed calendars for trading day, weekend/holiday closed day, and unknown calendar handling.

### Expected Output
A tested calendar abstraction exists and can be used by later worker workflows.

### Acceptance Criteria
- Calendar callers receive a normalized result for expected trading day, market closed, and unknown/unconfigured calendar.
- The abstraction accepts exchange and local date inputs.
- Tests use stubs/fakes rather than relying on live external calendars.
- The interface leaves room for later validation overrides and custom adapters.

### Test Requirements
Test expected trading day, closed day, and missing/unknown calendar responses.

### Dependencies
T3.

### Stop Condition
Stop when the calendar abstraction is tested and not wired into workflow automation.
