## T15: Health Check Endpoints and Worker Heartbeat Reporting

### Purpose
Expose foundational health checks for API, worker, database, scheduler, queue, and backup placeholder state.

### Scope
Add internal health endpoints or views plus worker/scheduler heartbeat records or read functions needed to report health status.

### Allowed Changes
- Health-check modules/endpoints
- Minimal heartbeat model or fields if not already covered by job records
- Health tests

### Forbidden Changes
- Do not implement alerting dashboards.
- Do not implement real backup jobs.
- Do not implement scheduled ingestion or worker queue execution.
- Do not expose sensitive internal details publicly.

### Inputs
- T2 database connectivity
- T8 load job fields for worker heartbeat/queue age where applicable
- Architecture health-check requirements

### Relevant Context
Milestone 1 acceptance requires health checks for API, worker heartbeat, database connectivity, scheduler heartbeat, queue depth or pending job age, and backup placeholder state. These checks support later single-owner operation.

### Implementation Budget
- Production files: expected maximum 2-4 closely related health/heartbeat files.
- Behavior count: 1 behavior, report current platform health.
- Model/API/workflow count: 0-1 heartbeat support model, 1 internal health API surface, 0 workflows.
- Test scope: healthy/unhealthy responses for API alive, database unavailable, missing/stale heartbeat, queue age/depth, and backup placeholder.

### Expected Output
An internal health surface reports platform components without running ingestion.

### Acceptance Criteria
- API liveness is reported.
- Database connectivity is checked.
- Worker and scheduler heartbeat status can be reported.
- Queue depth or oldest pending job age is reported from job records.
- Backup placeholder state is represented until real backups are implemented later.

### Test Requirements
Test health response shape and representative healthy/unhealthy states with controlled data or mocks.

### Dependencies
T2, T8.

### Stop Condition
Stop when health checks are tested and no real ingestion, backup, or alerting behavior has been added.

### Status
Done (2026-05-01)

- Added internal `GET /operator/health` platform health report with API, database, worker heartbeat, scheduler heartbeat, queue, and backup placeholder checks.
- Added runtime heartbeat persistence for `worker` and `scheduler` roles and best-effort heartbeat updates from worker ticks.
- Added targeted health reporting tests for healthy shape, missing/stale heartbeat and queue age, and database-unavailable behavior.
