## T010: Read Model Placeholder Tables

### Purpose
Create rebuildable read-model storage boundaries for future app and operator responses.

### Scope
Implement minimal model placeholders, migrations, admin registration, and tests for published app/operator read models needed by later milestones.

### Allowed Changes
- `project/apps/readmodels/`
- `project/apps/readmodels/migrations/`
- `project/apps/readmodels/tests/`
- `project/config/settings/` for app registration only

### Forbidden Changes
- Do not implement read-model rebuild jobs.
- Do not add public mobile API endpoints.
- Do not duplicate full source-table state unnecessarily.
- Do not introduce Redis or another cache.

### Inputs
- T007 exchange-day load models
- T008 publication models
- T009 signal storage models

### Relevant Context
The architecture starts with PostgreSQL-backed precomputed read views rather than Redis. Read models support exchange readiness, current-day prices, recent history, current/recent signals, and operator load summaries, and must be rebuildable from source tables.

### Expected Output
Database-backed read-model placeholders exist with clear ownership and enough fields to support future publication and app-read tasks.

### Acceptance Criteria
- Read model records reference source publication/load state where applicable.
- Read models include freshness/build timestamps.
- App-facing read models can be marked published/unpublished or active/inactive.
- Tests verify required source references and uniqueness by exchange/date/view type where applicable.

### Test Requirements
- Model tests for uniqueness and source references.
- Test that read-model app registration does not depend on Redis.

### Dependencies
T007, T008, T009.

### Stop Condition
Stop when placeholder read models exist without materializing real payloads.
