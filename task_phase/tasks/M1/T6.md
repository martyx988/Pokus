## T6: Implement Job State Transition and Idempotency Service

### Purpose
Provide a small domain service for safe, testable job state transitions and idempotency-key generation.

### Scope
- Add a service for creating or finding jobs by idempotency key.
- Add explicit transition functions for queued, running, retry_wait, succeeded, failed, cancelled, and stale_abandoned.
- Enforce terminal-state semantics for publication-related logic.
- Add lock/heartbeat timestamp updates required by later worker recovery.

### Allowed Changes
- `project/apps/domain/services/jobs.py`
- `project/apps/domain/selectors/jobs.py` if selectors are used by local pattern.
- `project/apps/domain/tests/test_jobs*.py`
- Minimal changes to `LoadJob` model methods only if needed to keep invariants close to the model.

### Forbidden Changes
- Do not implement Django-Q2 job scheduling.
- Do not implement external provider calls or ingestion.
- Do not add operator UI actions.
- Do not bypass database constraints added in T5.
- Do not make publication readiness decisions beyond exposing terminal-state predicates.

### Inputs
- T5 job schema.
- `design_phase/architecture.md` Worker/Scheduler Process and Job Failures.
- `task_phase/roadmap_plan.md` Milestone 1 validation and testing goals.

### Relevant Context
Weak job-state semantics are a key Milestone 1 risk. The worker later needs bounded retries, idempotency keys, heartbeat/lock expiry, and operator-visible manual outcomes, but this task should only define safe state behavior.

### Expected Output
- Job creation by idempotency key is deterministic.
- Invalid state transitions fail clearly.
- Terminal-state checks are reusable and tested.
- Heartbeat and lock metadata can be updated without changing job meaning.

### Acceptance Criteria
- Unit tests cover valid transitions from queued to running, retry_wait, succeeded, failed, cancelled, and stale_abandoned where applicable.
- Unit tests reject transitions out of terminal states.
- Unit tests verify repeated create/find with the same idempotency key returns one durable job.
- Unit tests verify terminal states for publication logic are exactly succeeded, failed, and cancelled.

### Test Requirements
- Unit tests for transition matrix.
- Unit tests for idempotency-key generation by exchange/day/instrument/load type.
- Unit tests for heartbeat update behavior.

### Dependencies
T5.

### Stop Condition
Stop when implementation agents can use a tested service for job state changes instead of writing ad hoc status mutations.
