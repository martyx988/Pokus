## T006: Load Job State Model and Transition Rules

### Purpose
Implement the persistent job-state foundation used by worker, publication, and operator workflows.

### Scope
Create `LoadJob` and focused domain helpers for allowed states, terminal-state semantics, idempotency keys, active-job locking metadata, attempts, timing, and failure reason storage.

### Allowed Changes
- `project/apps/jobs/`
- `project/apps/domain/models/` only if the project keeps job models there
- `project/apps/jobs/tests/`
- `project/apps/domain/tests/` only for direct job-model tests
- `project/config/settings/` for app registration only

### Forbidden Changes
- Do not implement actual background job execution or scheduling.
- Do not add ingestion, provider calls, signal generation, or publication decisions.
- Do not add operator manual action endpoints.
- Do not change non-job domain models except for relationships required by `LoadJob`.

### Inputs
- T003 core identity models
- T002 runtime-role configuration

### Relevant Context
Worker jobs must use explicit states: `queued`, `running`, `retry_wait`, `succeeded`, `failed`, `cancelled`, and `stale_abandoned`. Only `succeeded`, `failed`, and `cancelled` are terminal for publication-readiness logic. Jobs must be idempotent by exchange/day/instrument/load type where applicable.

### Expected Output
The database and domain helper layer can represent job state transitions safely before any real job logic is added.

### Acceptance Criteria
- `LoadJob` has the required states and terminal-state helper.
- Invalid state transitions are rejected by tests or domain validation.
- Idempotency key fields support exchange/date/load type and optional instrument/listing scope.
- Active duplicate jobs for the same idempotency key are prevented.
- Terminal jobs cannot be moved back to active states without an explicit future manual-action workflow.

### Test Requirements
- Unit tests for valid and invalid state transitions.
- Unit tests for terminal-state semantics.
- Unit tests for idempotency-key uniqueness and duplicate active-job prevention.

### Dependencies
T002, T003.

### Stop Condition
Stop when job records and transition tests pass without executing jobs.
