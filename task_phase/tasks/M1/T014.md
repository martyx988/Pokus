## T014: Health Check Endpoints and Worker Heartbeat State

### Purpose
Expose internal health signals for API, worker, database, scheduler, queue, and backup placeholder state.

### Scope
Implement health-check views/services and the minimal heartbeat state needed to report worker/scheduler liveness.

### Allowed Changes
- `project/apps/health/`
- `project/apps/jobs/` only for queue/heartbeat read helpers
- `project/config/urls.py`
- `project/config/settings/`
- `project/apps/health/tests/`
- `project/apps/jobs/tests/`

### Forbidden Changes
- Do not implement real scheduler loops or worker execution.
- Do not perform real backups.
- Do not expose health endpoints as public product APIs.
- Do not add external monitoring vendors.

### Inputs
- T002 runtime-role configuration
- T006 load job model

### Relevant Context
Milestone 1 must report API, worker, database, scheduler, queue age/depth, and backup placeholder state. Health checks support the VPS-first deployment and operator-focused reliability model.

### Expected Output
Internal health endpoints/services return structured status for core runtime dependencies and placeholders.

### Acceptance Criteria
- API liveness health can return healthy without database access.
- Database health verifies PostgreSQL connectivity.
- Worker heartbeat and scheduler heartbeat can be recorded and read.
- Queue health reports depth or oldest pending job age from persistent job records.
- Backup health reports placeholder state and last-known timestamps when available.
- Tests cover healthy and degraded database/heartbeat/queue conditions using controlled fixtures or mocks.

### Test Requirements
- Unit tests for health service status calculation.
- API tests for internal health responses.
- Tests for stale/missing heartbeat reporting.

### Dependencies
T002, T006.

### Stop Condition
Stop when health checks report structured status without running real ingestion or backups.
