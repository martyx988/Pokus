## T10: Add Health Check API for API, Worker, Database, Scheduler, Queue, and Backup Placeholder

### Purpose
Expose internal health checks required to prove the API, worker, database, scheduler, queue, and backup placeholder state are observable.

### Scope
- Add internal health endpoints for the API process.
- Add database connectivity checks.
- Add worker heartbeat and scheduler heartbeat checks.
- Add queue depth or pending job age reporting from T7 selectors.
- Add a backup placeholder state check using timestamp/config fields only.

### Allowed Changes
- `project/apps/operations/health.py`
- `project/apps/operations/views.py`
- `project/apps/operations/serializers.py`
- `project/apps/operations/urls.py`
- `project/config/urls.py`
- Tests under `project/apps/operations/tests/**`
- Minimal model/config changes for backup placeholder timestamp if no existing storage fits.

### Forbidden Changes
- Do not implement actual backup execution.
- Do not expose health endpoints as unauthenticated public mobile APIs.
- Do not include provider credentials, database passwords, or secret values in health responses.
- Do not implement operator dashboard UI.
- Do not add external monitoring vendors.

### Inputs
- T7 worker and scheduler heartbeat persistence.
- T9 private authentication boundary.
- `design_phase/architecture.md` Observability and Health Checks.
- `task_phase/roadmap_plan.md` Milestone 1 health acceptance criteria.

### Relevant Context
Milestone 1 requires health checks for API, worker heartbeat, database connectivity, scheduler heartbeat, queue depth or pending job age, and backup placeholder state. Backup execution is later infrastructure work, but status must have a placeholder surface.

### Expected Output
- A private/internal health endpoint returns structured component status.
- Component failures are represented clearly without leaking secrets.
- Health checks can be tested with deterministic database records.

### Acceptance Criteria
- Tests verify API alive status is returned.
- Tests verify database connectivity status is returned.
- Tests verify worker and scheduler heartbeat freshness is reported.
- Tests verify queue depth or pending job age is reported.
- Tests verify backup placeholder state is present.
- Tests verify unauthorized requests cannot read private health data.

### Test Requirements
- API integration tests for health endpoint success and unauthorized access.
- Unit tests for component status calculations.
- Database-backed tests for heartbeat and queue state.

### Dependencies
T7, T9.

### Stop Condition
Stop when a private health endpoint exposes every Milestone 1 health component in structured form.
