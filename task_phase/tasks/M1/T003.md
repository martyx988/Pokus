## T003: Core Identity Domain Models

### Purpose
Create the foundational identity schema for exchanges, instrument types, canonical instruments, listings, and stable identifiers.

### Scope
Implement only the model fields, constraints, migrations, and focused tests for:
- `Exchange`
- `InstrumentType`
- `Instrument`
- `Listing`
- `IdentifierRecord`

### Allowed Changes
- `project/apps/domain/models/`
- `project/apps/domain/admin.py`
- `project/apps/domain/migrations/`
- `project/apps/domain/tests/`

### Forbidden Changes
- Do not add support-state, universe-change, price, provider, job, signal, or publication models.
- Do not implement discovery or primary/best-listing ranking logic.
- Do not create public API serializers or views.
- Do not add provider-specific fields beyond generic identifier references.

### Inputs
- T001
- Architecture data model overview
- Product specification key entities and relationships

### Relevant Context
The product exposes only supported instruments, but the foundation must represent canonical instruments and exchange-specific listings. Initial instrument types are stocks, ETFs, and ETNs; crypto is deferred but the model should not block a future synthetic `CRY` exchange.

### Expected Output
The database can represent exchanges, instrument types, canonical instruments, listings, and provider/reference identifiers with relational integrity.

### Acceptance Criteria
- Migrations create all five model families.
- Exchange codes and instrument type codes are unique.
- Listings reference both an instrument and an exchange.
- Identifier records can attach stable provider/reference identifiers to instruments or listings.
- Tests cover uniqueness and required relationships.

### Test Requirements
- Model tests for required fields, uniqueness, and foreign-key relationships.
- Migration creation and application in the test database.

### Dependencies
T001.

### Stop Condition
Stop when identity models and tests pass without adding lifecycle or data-loading behavior.
