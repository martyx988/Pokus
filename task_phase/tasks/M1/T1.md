## T1: Backend Project Skeleton and Runtime Roles

### Purpose
Create the initial backend project structure with explicit API/web and worker/scheduler runtime roles.

### Scope
Set up the project source tree, dependency metadata, environment-loading pattern, and separate executable entrypoints for the API and worker roles.

### Allowed Changes
- `project/` backend scaffold files
- Dependency/configuration files needed to run the backend locally
- Minimal local documentation for starting API and worker roles

### Forbidden Changes
- Do not create domain models, migrations, ingestion logic, provider adapters, or app-facing business endpoints.
- Do not introduce Redis, distributed services, or non-PostgreSQL persistence.
- Do not implement production deployment hardening beyond the local skeleton.

### Inputs
- `task_phase/roadmap_plan.md`
- `design_phase/architecture.md`
- `design_phase/product_spec.md`

### Relevant Context
The architecture is a modular monolith: one codebase, separate API/web and worker/scheduler processes, and one shared PostgreSQL database. The API serves mobile and private operator/admin reads; the worker owns scheduled and asynchronous backend work.

### Implementation Budget
- Production files: expected maximum 6-10 scaffold/config/entrypoint files.
- Behavior count: 1 behavior, running distinct API and worker process roles from the same codebase.
- Model/API/workflow count: 0 domain models, 0 product APIs, 2 runtime entrypoints.
- Test scope: smoke tests or checks that import/start the configured roles without domain behavior.

### Expected Output
A runnable backend skeleton exists under `project/` with clear API and worker entrypoints and shared settings.

### Acceptance Criteria
- API/web and worker roles can be invoked independently.
- Both roles use shared project settings and module layout.
- The source layout makes room for domain, jobs, calendars, API, worker, observability, and admin modules without implementing feature workflows.
- Local start instructions or commands are documented.

### Test Requirements
Run the available scaffold smoke checks or import tests for both entrypoints.

### Dependencies
None.

### Stop Condition
Stop when the project skeleton and role entrypoints exist and no domain behavior has been added.
