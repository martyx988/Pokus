## T1: Create Django Project Skeleton and Runtime Roles

### Purpose
Create the deployable project foundation for a modular Django monolith with distinct API/web and worker/scheduler runtime roles.

### Scope
- Initialize the `project/` source tree.
- Add Django and Django REST Framework dependency metadata.
- Add the base Django settings package and URL entrypoint.
- Add separate API and worker command entrypoints that can start without implementing ingestion.
- Add empty first-party app/module boundaries for domain, operations, API, and worker code.

### Allowed Changes
- `project/pyproject.toml`
- `project/manage.py`
- `project/config/**`
- `project/apps/**/__init__.py`
- `project/apps/**/apps.py`
- `project/README.md`
- Minimal test files required to prove the project imports.

### Forbidden Changes
- Do not create market-data provider integrations.
- Do not implement ingestion, publication, signal computation, universe discovery, or admin workflows.
- Do not add non-Django backend frameworks.
- Do not add Redis, Kafka, RabbitMQ, or other external queue infrastructure.
- Do not create database tables beyond Django's required built-in app migrations in this task.

### Inputs
- `task_phase/roadmap_plan.md` Milestone 1.
- `design_phase/architecture.md` sections on the recommended architecture, API/web process, worker/scheduler process, and backend runtime.
- `design_phase/product_spec.md` launch scope and system constraints.

### Relevant Context
The architecture requires a Django with Django REST Framework modular monolith. API/web and worker/scheduler are separate runtime roles sharing one codebase and one PostgreSQL database. The worker must own background work later, while the API must serve public mobile and private operator/admin surfaces.

### Expected Output
- A bootstrapped Django project exists under `project/`.
- API/web startup and worker/scheduler startup are represented as separate commands or documented entrypoints.
- First-party module boundaries exist but contain no real domain behavior yet.

### Acceptance Criteria
- `python manage.py check` succeeds from `project/`.
- The Django settings module loads without requiring provider keys or production-only secrets.
- API and worker roles are distinguishable by command, environment variable, or documented invocation.
- The source tree clearly separates API, worker, domain, and operations modules.

### Test Requirements
- Add a smoke test that imports the Django settings.
- Add a smoke test that validates the project URL configuration can be loaded.
- Add a smoke test or documented command check for the worker entrypoint.

### Dependencies
None.

### Stop Condition
Stop when the Django project skeleton can be checked locally and the runtime-role boundaries are visible without implementing domain features.
