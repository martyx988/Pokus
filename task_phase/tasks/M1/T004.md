## T004: Supported Universe State and Change History Models

### Purpose
Represent current app visibility and lightweight audit history for supported-universe changes.

### Scope
Implement model fields, constraints, migrations, admin registration, and tests for:
- `SupportedUniverseState`
- `UniverseChangeRecord`

### Allowed Changes
- `project/apps/domain/models/`
- `project/apps/domain/admin.py`
- `project/apps/domain/migrations/`
- `project/apps/domain/tests/`

### Forbidden Changes
- Do not implement automatic discovery, exclusion calculations, delisting detection, or restoration workflows.
- Do not add public supported-universe API endpoints.
- Do not modify provider, price, job, signal, or publication models.
- Do not decide future exchange expansion policy.

### Inputs
- T003 core identity models

### Relevant Context
The app may see only the final supported universe. The system must preserve additions, removals, exclusions, delisting suspicion, degradation, restoration, and symbol/name/identifier changes as lightweight historical records with explicit reasons.

### Expected Output
The database can store one current support-state record for a chosen listing and append universe-change audit records.

### Acceptance Criteria
- Supported state references a listing and records price availability and signal readiness separately.
- Historical-only/inactive states can be represented without deleting instrument history.
- Universe-change records include effective day, event type, reason, old state, new state, and details.
- Tests enforce the one-current-state-per-listing rule.

### Test Requirements
- Model tests for supported, inactive, historical-only, and signal-not-ready states.
- Model tests for allowed universe-change event types and required reason/details fields.

### Dependencies
T003.

### Stop Condition
Stop when support state and audit history exist without implementing lifecycle automation.
