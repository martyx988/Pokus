## T007: Exchange-Day Load and Instrument Outcome Models

### Purpose
Represent aggregate exchange/day load state and per-instrument terminal outcomes for later publication checks.

### Scope
Implement model fields, constraints, migrations, admin registration, and tests for:
- `ExchangeDayLoad`
- `InstrumentLoadOutcome`

### Allowed Changes
- `project/apps/domain/models/`
- `project/apps/domain/admin.py`
- `project/apps/domain/migrations/`
- `project/apps/domain/tests/`

### Forbidden Changes
- Do not implement publication readiness calculation.
- Do not implement coverage calculation beyond storing fields needed later.
- Do not execute jobs or call providers.
- Do not add operator dashboard endpoints.

### Inputs
- T003 core identity models
- T005 price storage models
- T006 load job model

### Relevant Context
Publication depends on all eligible instruments reaching terminal load outcomes for an exchange/day/load type. Exchange/day statuses must be able to represent not started, in progress, market closed, partial/problematic, ready, and failed.

### Expected Output
The schema can track load progress per exchange/day/load type and final per-instrument results.

### Acceptance Criteria
- `ExchangeDayLoad` references exchange, trading date, load type, status, timing, counts, coverage placeholder, degraded flag, and exception count.
- `InstrumentLoadOutcome` references exchange-day load, listing, optional price record, outcome status, failure reason, and terminal timestamp.
- A uniqueness constraint prevents duplicate outcomes for one exchange-day load and listing.
- Tests cover allowed statuses and outcome uniqueness.

### Test Requirements
- Model tests for exchange-day statuses.
- Model tests for terminal and non-terminal instrument outcomes.
- Model tests for unique outcome per listing per exchange-day load.

### Dependencies
T003, T005, T006.

### Stop Condition
Stop when load/outcome state can be stored without calculating readiness.
