## T001: Django Project Skeleton

### Purpose
Create the initial Django/DRF codebase structure that later milestone tasks can extend.

### Scope
Set up the project package, app package layout, dependency metadata, local environment example, and a minimal test harness.

### Allowed Changes
- `project/pyproject.toml`
- `project/manage.py`
- `project/config/`
- `project/apps/`
- `project/tests/`
- `project/.env.example`
- `project/README.md`

### Forbidden Changes
- Do not add domain models beyond empty app scaffolding.
- Do not add ingestion, signal, publication, or provider logic.
- Do not create app-facing business endpoints other than framework health placeholders if needed for startup.
- Do not modify files outside `project/`.

### Inputs
- `task_phase/roadmap_plan.md`
- `design_phase/architecture.md`
- `design_phase/product_spec.md`

### Relevant Context
The architecture confirms Django with Django REST Framework as the initial backend. The system is a modular monolith with one shared codebase and separate API/web and worker/scheduler roles. PostgreSQL is the source of truth; Redis is not part of launch scope.

### Expected Output
A runnable Django project skeleton exists under `project/` with clear app/module boundaries for domain, API, worker, operations, and shared configuration.

### Acceptance Criteria
- `manage.py` and Django settings can import successfully.
- The project includes DRF and PostgreSQL-ready dependencies.
- The app layout makes API/web and worker code share the same domain modules.
- A minimal test command can run and discover tests.
- Environment variable names are documented in `.env.example` without real secrets.

### Test Requirements
- Add a smoke test proving Django settings load.
- Run the repository's chosen test command from `project/`.

### Dependencies
None.

### Stop Condition
Stop when the skeleton runs locally and no domain behavior has been implemented.
