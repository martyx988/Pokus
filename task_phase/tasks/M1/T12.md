## T12: Add Milestone 1 End-to-End Validation Tests and Documentation

### Purpose
Prove the Milestone 1 foundation works as one deployable slice before later ingestion, publication, and signal work begins.

### Scope
- Add integration tests for API/worker/database connectivity.
- Add migration apply test coverage or a documented migration validation command.
- Add health endpoint integration coverage against PostgreSQL-backed state.
- Add concise documentation for running the API, worker, migrations, tests, and health checks locally.

### Allowed Changes
- `project/tests/**`
- Existing app test files only to add cross-component integration coverage.
- `project/README.md`
- `project/docker-compose.yml` only for test command wiring if necessary.
- CI configuration if the repository already has one; otherwise do not introduce CI unless trivial and local-only.

### Forbidden Changes
- Do not implement new product behavior.
- Do not broaden schema beyond prior Milestone 1 tasks.
- Do not add future milestone endpoints.
- Do not require paid services, external providers, or production secrets for tests.
- Do not mask failing tests by weakening acceptance checks from prior tasks.

### Inputs
- T1 through T11.
- `task_phase/roadmap_plan.md` Milestone 1 Validation Goal, Acceptance Criteria, and Testing Strategy.
- `design_phase/architecture.md` Deployment / Hosting, Health Checks, and Data Model Overview.

### Relevant Context
The validation goal is to prove API and worker processes share durable state, create and transition jobs idempotently, expose health, and represent exchange/day lifecycle states without real provider ingestion.

### Expected Output
- A single documented local path validates the Milestone 1 foundation.
- Tests cover cross-role database sharing, health checks, job transitions, and migration creation.
- README instructions are enough for a future implementation agent to run and verify the foundation.

### Acceptance Criteria
- Integration tests prove API and worker code can access the same PostgreSQL database.
- Integration tests prove health checks report API, worker, database, scheduler, queue, and backup placeholder state.
- Existing unit tests for job state, idempotency, and domain constraints pass.
- Documentation lists exact commands for migrations, tests, API startup, worker startup, and health check access.

### Test Requirements
- Run the full test suite.
- Run Django system checks.
- Run migrations on a clean PostgreSQL database.
- If rollback is supported by the chosen migration/test tooling, validate rollback for initial migrations or document why rollback validation is not automated.

### Dependencies
T1, T2, T3, T4, T5, T6, T7, T8, T9, T10, T11.

### Stop Condition
Stop when Milestone 1 can be validated end to end using documented local commands and no external market-data providers.
