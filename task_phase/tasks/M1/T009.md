## T009: Signal Storage Models

### Purpose
Reserve durable boundaries for signal statistics and backend-generated signal events.

### Scope
Implement model fields, constraints, migrations, admin registration, and tests for:
- `SignalStatistic`
- `SignalEvent`

### Allowed Changes
- `project/apps/domain/models/`
- `project/apps/domain/admin.py`
- `project/apps/domain/migrations/`
- `project/apps/domain/tests/`

### Forbidden Changes
- Do not implement or port the alert algorithm.
- Do not generate historical or current signals.
- Do not expose signal API endpoints.
- Do not rewrite price-storage or publication models.

### Inputs
- T003 core identity models
- T005 price storage models

### Relevant Context
The backend computes only `Dip` and `Skyrocket` signals using an existing alert algorithm. Signal generation must distinguish no signal from unavailable due to insufficient history. Post-launch signal records are not rewritten retrospectively due to algorithm changes.

### Expected Output
The schema can store supporting statistics and generated signal events for a listing/date without computing them.

### Acceptance Criteria
- `SignalStatistic` links to listing and trading date and stores structured statistic values/context.
- `SignalEvent` supports `dip` and `skyrocket` event types and references relevant price/statistic records where practical.
- Uniqueness constraints prevent duplicate signal events for the same listing/date/type/algorithm version.
- Tests cover supported event types and duplicate prevention.

### Test Requirements
- Model tests for signal event choices.
- Model tests for algorithm-version-aware uniqueness.
- Model tests for required listing/date relationships.

### Dependencies
T003, T005.

### Stop Condition
Stop when signal storage exists without computing or publishing signals.
