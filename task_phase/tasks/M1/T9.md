## T9: Exchange-Day Load and Instrument Outcome Schema

### Purpose
Create load-state records that connect jobs, exchange/day units, and per-instrument outcomes.

### Scope
Add `ExchangeDayLoad` and `InstrumentLoadOutcome` models/migrations for daily opening and historical close load types, aggregate statuses, counts, timing, and per-instrument terminal outcomes.

### Allowed Changes
- Domain model files for load aggregates and outcomes
- Migrations for load/outcome tables
- Focused model tests

### Forbidden Changes
- Do not implement publication coverage calculation.
- Do not implement ingestion or provider retry execution.
- Do not implement operator load tables.

### Inputs
- T3 exchange schema
- T4 listing/support schema
- T8 load job schema and terminal-state semantics

### Relevant Context
Publication later depends on terminal outcomes across all eligible instruments for an exchange/day. Operator views later need exchange/day status, timing, eligible count, success/failure count, coverage, quality result, degraded flag, and exceptions.

### Implementation Budget
- Production files: expected maximum 1-3 closely related domain/migration files.
- Behavior count: 1 behavior, durable load aggregate and per-instrument outcome representation.
- Model/API/workflow count: 2 model entities, 0 APIs, 0 workflows.
- Test scope: load type/status constraints, outcome uniqueness, job references, and count/timing fields.

### Expected Output
Exchange/day loads and instrument load outcomes can be stored for later publication and operator visibility.

### Acceptance Criteria
- Load types distinguish current-day opening-price loads from historical adjusted-close loads.
- Exchange/day load statuses can represent not started, in progress, market closed, partial/problematic, ready, and failed.
- Instrument outcomes can represent succeeded, failed, cancelled, pending/in-progress where appropriate for pre-terminal work.
- Outcome uniqueness prevents duplicate records for the same listing, exchange/day, and load type.

### Test Requirements
Test valid statuses/load types, uniqueness, and relationships to exchange, listing, and job records.

### Dependencies
T4, T8.

### Stop Condition
Stop when load aggregate/outcome schema exists and is tested without publication logic.
