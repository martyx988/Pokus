## T9: Add Public App Credential and Private Admin Authentication Boundary

### Purpose
Establish the basic access-control boundary between public mobile reads and private operator/admin surfaces.

### Scope
- Add app-level credential or signed-token validation middleware/permission classes for public API identification.
- Keep Django authenticated sessions for operator/admin users.
- Add role/group checks for operator read access and admin command access.
- Add credential rotation/revocation-ready storage if database-backed credentials are chosen.

### Allowed Changes
- `project/apps/api/authentication.py`
- `project/apps/api/permissions.py`
- `project/apps/operations/permissions.py`
- `project/config/settings/**`
- `project/config/urls.py`
- Tests under `project/apps/api/tests/**` and `project/apps/operations/tests/**`
- Optional domain model/migration for app credentials.

### Forbidden Changes
- Do not implement end-user account ownership or user-specific alert logic.
- Do not expose admin actions publicly.
- Do not store real mobile credentials or signing secrets in source control.
- Do not implement rate limiting unless a minimal hook is required by the chosen authentication class.
- Do not create business endpoints beyond authentication test fixtures.

### Inputs
- T1 project skeleton.
- `design_phase/architecture.md` Authentication / Session Model and Security and Access Control.
- `design_phase/product_spec.md` system constraints and API-level behavior.

### Relevant Context
The backend does not own user-specific alert state. Mobile credentials are app identification and throttling controls rather than strong user authentication. Operator/admin access is private and session-authenticated, with admin actions audited later.

### Expected Output
- Public API views can require app-level credentials.
- Operator/admin views can require authenticated users and role-specific permissions.
- Secrets are read from environment variables or database placeholders, not hardcoded.

### Acceptance Criteria
- Tests verify public API test view rejects missing/invalid app credential.
- Tests verify public API test view accepts a configured valid credential.
- Tests verify operator-only access requires authenticated operator/admin role.
- Tests verify admin-only access requires admin role.

### Test Requirements
- Authentication unit tests.
- Permission tests for anonymous, mobile credential, operator, and admin identities.
- Settings test ensuring no default production secret is embedded.

### Dependencies
T1.

### Stop Condition
Stop when implementation agents have reusable authentication/permission primitives for future public and private endpoints.
