## T43: Schedule Exchange-Day Opening Load Jobs

### Purpose
Create the scheduler path that enqueues current-day opening-load jobs per exchange/day with idempotent keys and lock-safe execution entry.

### Scope
Add scheduling and enqueue behavior only for today opening-price load jobs. No provider fetch or publication logic.

### Allowed Changes
- Worker scheduling/queue modules for daily opening-load job creation
- Job identity/idempotency helpers used for exchange/day opening jobs
- Focused tests for enqueue deduplication and lock-safe scheduling

### Forbidden Changes
- Do not implement provider fetch or candidate selection behavior.
- Do not compute coverage or publication readiness.
- Do not expose API endpoints.

### Inputs
- Milestone 1 job state and locking model
- Milestone 2 supported exchanges and trading-day calendar checks
- `design_phase/product_spec.md` FR-55c

### Relevant Context
Milestone 4 starts with reliable daily job creation. Exchange/day publication depends on complete per-exchange execution, so scheduling must be deterministic, idempotent, and calendar-aware.

### Implementation Budget
- Production files: max 3-4 worker/job scheduling files.
- Behavior count: 1 behavior (schedule and enqueue opening-load jobs).
- Model/API/workflow count: 0 new models, 0 APIs, 1 job-scheduling workflow.
- Test scope: enqueue idempotency, expected-trading-day scheduling, lock-safe duplicate prevention.

### Expected Output
Worker can enqueue one opening-load workflow per eligible exchange/day without duplicate job creation.

### Acceptance Criteria
- Scheduler creates opening-load jobs only for configured supported exchanges on expected trading days.
- Repeated scheduler runs do not create duplicate opening-load jobs for the same exchange/day.
- Job records use existing lock/idempotency patterns compatible with retries and stale-recovery flows.

### Test Requirements
- Tests for deduplicated enqueue behavior.
- Tests for market-closed skip behavior.
- Tests for deterministic idempotency key composition.

### Dependencies

### Stop Condition
Stop when opening-load jobs are reliably scheduled and deduplicated, without implementing downstream load logic.
